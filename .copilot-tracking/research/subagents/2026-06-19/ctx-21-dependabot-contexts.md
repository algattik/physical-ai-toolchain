# 21 Dependabot Contexts

```yaml
  PYTHON: /workspace/isaaclab/isaaclab.sh -p
  ACCEPT_EULA: "Y"
  PRIVACY_CONSENT: "Y"
  HYDRA_FULL_ERROR: "1"

  # Training reads $AZURE_ML_OUTPUT_<NAME> directly from the runtime.
  # The following are set by submit-azureml-training.sh via --set flags
  # AZURE_SUBSCRIPTION_ID: <set by script>
```

Caption: This is not a toy YAML file; it is the Azure ML job contract for an Isaac Lab GPU training container, including the Isaac runtime wrapper, NVIDIA/EULA requirements, and checkpoint upload behavior (`training/rl/workflows/azureml/train.yaml:53-63`).

## Generalist Explanation

The 21 contexts exist because this monorepo ships several independent runtime stacks, not one application with one lock file. A Dependabot bump can target a React UI, a FastAPI API, Terraform providers, GitHub Actions, Go contract tests, Azure ML training, OSMO pods, or a GPU evaluation image. That creates high update volume because each ecosystem has its own cadence and transitive graph. It also creates high blast radius: a safe-looking Python bump in one context can break a CUDA, Torch, Isaac Sim, LeRobot, or Azure ML ABI that the other contexts never exercise. The ignored and overridden dependencies are evidence of real runtime constraints: Torch/CUDA wheels, Isaac container Python, Azure ML SDK caps, LeRobot metadata, offline model weights, and security-patched transitive packages all have to line up at the same time.
